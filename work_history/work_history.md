### 작업 배경
- 로컬 환경에서는 3분리 입력 필드 휴대폰번호(`[010]`-`[국번]`-`[끝번호]`) 마스킹이 정상 동작했으나, **운영서버 배포 후 마스킹이 되지 않는 문제** 발생
- OCR 판독 시 대괄호로 감싸진 형태(`[010]`, `[3559]`, `[4313]`)로 인식되는 분리 입력 필드의 실제 값이 마스킹 처리되지 않음
- 마스킹 항목에 **차량번호** 추가 요청

### 버그 1: 분리 입력 필드 전화번호 마스킹 미동작 (운영서버)

#### 원인 분석

**원인 1 — 끝번호 필드 마스킹 누락 (핵심 버그)**
- 기존 코드(`detect_personal_info` 내 "2-a. 분리된 전화번호 입력 필드 탐지" 섹션)는
  `[010]` - `[국번]` - `[끝번호]` 중에서 **국번(`seg2_word`)만 마스킹하고 끝번호(`seg3_word`)를 마스킹하지 않는 버그**가 존재
- 이로 인해 끝 4자리 실제 번호가 그대로 노출됨

**원인 2 — 세그먼트 간 간격 허용치 부족**
- 대괄호(`[ ]`) 포함 입력 필드는 일반 텍스트보다 단어 간격이 넓을 수 있음
- 기존 허용치: `height × 8.0` (최소 120px) → 운영서버 레이아웃에서 간격 초과로 탐지 실패

#### 수정 내용 (`masking_core.py` L1307~L1337)

```python
# 수정 전: 국번만 마스킹
mask_regions.append({
    'x': seg2_word['x'], 'y': seg2_word['y'],
    'width': seg2_word['width'], 'height': seg2_word['height']
})

# 수정 후: 국번 + 끝번호 모두 마스킹
mask_regions.append({
    'x': seg2_word['x'], 'y': seg2_word['y'],
    'width': seg2_word['width'], 'height': seg2_word['height']
})
mask_regions.append({
    'x': seg3_word['x'], 'y': seg3_word['y'],
    'width': seg3_word['width'], 'height': seg3_word['height']
})
```

- 세그먼트 간 간격 허용치 변경:  
  `height × 8.0 (min 120px)` → **`height × 15.0 (min 200px)`**

## 2026-06-15 (추가) — 캡쳐방지 우회 후 운영서버 마스킹 실패 근본 원인 수정

### 문제 제보
> "캡쳐방지 우회 작업 이전에는 휴대폰번호 마스킹이 잘 됐는데, 이후 운영서버에서 잘 안 됨"

### 근본 원인: Windows DPI Awareness 미설정으로 인한 좌표계 불일치

운영서버 PC가 **고DPI(125%, 150%)** 설정인 경우, DPI Awareness가 없으면:

| 구분 | 로컬(100% DPI) | 운영서버(125% DPI) |
|------|--------------|-------------------|
| Tkinter 창 좌표 | 논리px = 물리px | 논리px (작음) |
| `mss`/`dxcam` 캡쳐 이미지 | 물리px = 논리px | **물리px (1.25배 큰 이미지)** |
| OCR word.height | 예) 18px | 예) 22~27px |
| 분리필드 gap 허용치 | 18×8=144px ✅ | **22×8=176px → 실제 gap 초과로 탐지 실패** |

**흐름:**
1. `screen_grab.py`의 `mss`가 물리픽셀(큰 이미지)로 캡쳐
2. `capture_window.py`의 Tkinter 창은 논리픽셀 크기
3. 이미지가 창보다 커서 마우스 선택 영역이 오프셋됨
4. OCR 좌표도 물리픽셀 기준으로 나와 마스킹 영역이 실제 텍스트 위치와 어긋남

### 수정 내용 (`capture_window.py`)

```python
# 1. 모듈 임포트 시 즉시 DPI Awareness 설정 (Tkinter 창 생성 전)
_set_dpi_awareness()   # Per-Monitor V2 → V1 → 레거시 순 폴백

# 2. 캡쳐 이미지 크기 ≠ Tkinter 창 크기일 때 자동 리사이즈
img_w, img_h = self.screenshot.size
if img_w != width or img_h != height:
    print(f"[DPI보정] 캡쳐이미지({img_w}×{img_h}) → Tkinter창({width}×{height})으로 리사이즈")
    self.screenshot = self.screenshot.resize((width, height), Image.Resampling.LANCZOS)
```

**적용 우선순위:**
1. `SetProcessDpiAwareness(2)` — Per-Monitor DPI V2 (Windows 10 Anniversary+)
2. `SetProcessDpiAwareness(1)` — Per-Monitor DPI V1 (Windows 8.1+)  
3. `SetProcessDPIAware()` — 시스템 DPI (레거시 폴백)

### 수정 파일
- `capture_window.py` — DPI Awareness 설정 + 이미지 크기 보정

#### 1. 운전면허번호 포맷 수정

- **`rrn` 레이블 기반 분리 필드 주민번호 탐지**

## 2026-06-16 — 주민등록번호 및 운전면허번호 마스킹 누락 현상 수정

### 작업 배경
- 캡처된 화면 이미지 중 특정 형태(이름 옆에 배치된 주민등록번호 등) 및 일부 단어로 쪼개져 인식되는 운전면허번호의 마스킹이 누락되는 현상 해결
- 축소 이미지 분석을 통하여 오동작하는 경계 케이스(Edge Cases)를 식별하고 탐지 규칙의 강건성 보완

---

### 수정 파일
- `masking_core.py`

---

### 수정 내용

#### 1. 성명(`name`) 레이블 우측 주민번호 탐색 버그 수정
- **원인**: 성명 레이블 우측 후보들을 탐색할 때, 성명을 발견한 뒤(`name_found = True`) 뒤이은 문자열에 한글/영문자가 없으면(즉, 숫자로만 된 주민등록번호 필드) 스캔이 중단(`break`)되는 오동작 버그가 있었습니다.
- **수정**: 빈 문자열(`cand_clean == ""`, 한글/영문자 없음)을 만나더라도 `break`하지 않고 그냥 건너뜀(`pass`)으로써 주민번호(`RRN_PATTERN`) 매칭 검사가 정상적으로 수행될 수 있도록 흐름을 수정했습니다. 이로써 `[피보험자] [이준희] [731225-1542622]` 조합에서 이름과 주민등록번호가 모두 올바르게 마스킹됩니다.

#### 2. 운전면허번호(`driver`) 쪼개짐 수집 로직 개선
- **원인**: 운전면허번호 입력란이 `[92-692533] [-74]` 처럼 여러 개의 쪼개진 단어로 OCR 인식될 때, 개별 단어의 숫자 자릿수가 `(2, 6, 10)` 중 하나여야 한다는 엄격한 제약 때문에 첫 단어(`92-692533`, 8자리) 수집 시 루프가 즉시 중단되던 문제가 있었습니다.
- **수정**: 개별 단어의 자릿수 제한을 풀고 숫자가 들어간 단어들을 누적 합산하여 최종 10~12자리가 될 때까지 계속 수집하도록 변경했습니다.
- **추가 개선**: 쪼개진 첫 번째 단어 내에 지역코드(2자리) 외에 일련번호의 일부가 합쳐져서 인식된 경우(예: `92-692533`), 첫 단어 내에서도 앞 2자리(지역코드) 이후 부분(`-692533`)을 부분 마스킹 박스로 계산하여 개인정보 노출을 완벽하게 차단했습니다.

#### 3. 검증
- 가상 시나리오 및 실제 OCR 인식 형태 레이아웃 검증용 테스트 스크립트(`scratch/test_real_image.py`)를 작성하여 위의 모든 조건이 정상적으로 작동하고 마스킹 영역이 바르게 검출되는 것을 입증했습니다.

- **효과**: 입력창 레이아웃이 쪼개져 있거나 줄바꿈이 일어난 상태에서도 상세주소 영역만 정확한 좌표에 마스킹 박스가 적용됩니다.

- 시스템 표준 `messagebox.showwarning` 다이얼로그는 특정 글자의 색상이나 서식(Bold 등)을 변경할 수 없는 운영체제 한계가 있었습니다.
- 사용자 요청에 맞게 핵심 키워드인 **"F9"** 텍스트를 붉은색 계열로 굵게 강조하고, 깔끔한 2줄 레이아웃으로 메시지를 렌더링하기 위해 Tkinter `Toplevel`을 기반으로 한 고유 디자인 모달 알림창을 설계하여 적용했습니다.

### 수정 내용

#### 1. 커스텀 모달 다이얼로그 개편 (`main.py`)
- **수정**: 기존 표준 팝업 대신 `tk.Toplevel`을 활용해 메인 윈도우 중앙에 자동 정렬되는 380x160 크기의 다이얼로그를 자체 렌더링합니다.
- **텍스트 포맷팅**:
  - `tk.Text` 위젯을 이용하여 배경과 테두리를 투명화한 뒤 라벨처럼 표시되도록 설정했습니다.
  - 서식 태그(`tag_configure`)를 이용하여 일반 글자는 검은색 계열(#202124)로 처리하고, 핵심 단어인 **"F9"** 키에 대해서는 구글 레드 계열(#D93025)과 굵은 폰트(bold) 스타일을 지정했습니다.
- **가독성 줄바꿈**:
  - 사용자가 요청하신 대로 문구를 2줄로 자연스럽게 정렬했습니다:
    ```
    타 캡쳐프로그램은 사용 불가 합니다.
    F9 키를 사용하여 개인정보 마스킹캡쳐를 이용해 주세요.
    ```
- **UI 스타일링**:
  - 다이얼로그 배경을 소프트 화이트(#F8F9FA)로 감싸고, 확인 버튼을 플랫한 윈도우 블루(#0078D7) 스타일 및 활성 상태 색상반응(#106EBE)과 마우스 핸드 커서(`hand2`)를 적용하여 완결성 높은 모던 UI를 구현했습니다.
